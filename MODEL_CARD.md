# Model Card: Bedrock Guardrail Graph Optimizer

## Summary

This is **not a trained machine learning model**. It is a graph-based, rule-driven optimizer that recommends guardrail placement across simulated Bedrock-style multi-agent workflows.

## What it is

- A directed-graph model of multi-agent workflow execution (agents, tools, user input, and final response as nodes; handoffs as edges).
- A transparent, rule-based classifier (see `config/guardrail_policy.yaml`) that recommends `KEEP`, `REUSE_PREVIOUS_DECISION`, `SKIP_REDUNDANT_WITH_AUDIT`, `MOVE_TO_BOUNDARY`, or `HUMAN_REVIEW_REQUIRED` for each guardrail check.
- A metrics and explanation layer that reports graph-level statistics, cost/latency savings estimates, and plain-language audit reasoning.

## What it is not

- Not a live integration with Amazon Bedrock Guardrails or AgentCore Policy APIs.
- Not a trained or fine-tuned model of any kind -- every decision traces back to an explicit, inspectable rule.
- Not a formally verified safety system.
- Not intended to remove, bypass, or weaken production security controls without human approval.

## Inputs

Synthetic Bedrock-style workflow traces (`data/synthetic_workflow_traces.csv`), generated by `src/synthetic_workflows.py`. These traces are fabricated for demonstration and testing; they do not contain real user data or real Bedrock telemetry.

## Outputs

- `outputs/graph_nodes.csv`, `outputs/graph_edges.csv` -- structural graph tables.
- `outputs/workflow_graph_summary.csv` -- per-workflow graph-level features and risk scores.
- `outputs/redundant_paths.csv` -- graph-path-aware duplicate detection results.
- `outputs/optimized_guardrail_plan.csv` -- the final per-step placement recommendation.
- `outputs/guardrail_metrics.json`, `outputs/boundary_coverage_matrix.csv` -- aggregate and safety metrics.
- `outputs/guardrail_explanations.json` -- human-readable audit explanations for top-impact workflows.

## Safety behavior

- Protected trust boundaries (`USER_TO_AGENT`, `AGENT_TO_TOOL`, `TOOL_TO_AGENT`, `AGENT_TO_FINAL_RESPONSE`) are always kept.
- High-risk checks are kept or escalated to human review -- never silently skipped.
- Text mutation, policy drift, and unsafe prior results (`WARN`/`BLOCK`) always block reuse.
- `graph_metrics.py` enforces a hard safety gate: protected-boundary coverage and high-risk preservation must both equal 100%, or the pipeline raises an error rather than shipping a plan silently.

## Intended use

Research, prototyping, and demonstration of graph-based guardrail placement reasoning for Bedrock-style multi-agent systems. It is a starting point for discussion with platform and security teams, not a drop-in production safety control.

## Out-of-scope use

- Do not use this to automatically remove or disable guardrails in a production system without human review.
- Do not treat simulated latency/cost figures as measured production numbers.
- Do not treat `graph_risk_score` as a certified or formally validated risk metric -- it is a transparent, explainable heuristic.
